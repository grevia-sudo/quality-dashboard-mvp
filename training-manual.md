# 回收品檢系統 MVP 教育訓練文檔

作者：**Manus AI**  
版本日期：2026-04-27

## 1. 文件目的

本文件供現場工程師、主管與管理者作為上線教育訓練與日常交接使用。內容聚焦於目前已落地的 **回收品檢系統 MVP**，說明系統入口、角色權限、各站點操作方式、管理後台工具，以及常見作業情境。本文以目前專案實作與頁面行為為基礎整理，包含首頁導流、站點總覽、匯入作業、A1/A2/B/C/D/E/待入庫流程、工程師 KPI 與管理後台等模組。[1] [2] [3] [4] [5] [6]

## 2. 系統定位與使用情境

回收品檢系統 MVP 是一套服務多站點回收檢驗流程的內部作業平台。系統將商品從匯入、到貨補齊、各站檢測、全檢確認到待入庫的流程串接起來，並同步提供工程師 KPI 與管理設定能力。首頁會依登入者角色，自動導向較適合的工作入口；一般工程師偏向進入站點總覽，管理角色則會被導向管理後台或相關總覽入口。[1] [2] [6]

| 模組 | 主要目的 | 主要使用者 |
| --- | --- | --- |
| 首頁與導流 | 依角色引導至站點總覽或管理後台 | 全體使用者 |
| 站點總覽 | 查看各站待辦、逾期與今日新增 | 工程師、主管、管理者 |
| 匯入作業 | 建立採購單與匯入待處理商品 | supervisor / manager / admin |
| A1/A2/B/C/E 站 | 執行站點檢測、補錄與推進下一站 | 工程師 |
| D 站全檢 | 帶入 B/C 結果做全檢複核與分流 | supervisor / manager / admin |
| 工程師 KPI | 查看個人當日與月度績效 | 全體登入使用者 |
| 管理後台 | KPI 分析、流程設定、資料工具、帳號管理 | admin |

## 3. 角色與權限說明

目前系統的管理型頁面採明確權限收斂。`supervisor`、`manager`、`admin` 可進入匯入作業與 D 站全檢；`admin` 才可進入管理後台；一般受保護流程則需要登入後才能使用。若使用者已登入但不具備管理型作業權限，系統會將其導回站點總覽，避免誤進入不應操作的頁面。[1] [5] [7] [8]

| 角色 | 可見／可操作模組 | 補充說明 |
| --- | --- | --- |
| user | 站點總覽、個人工程師 KPI、一般站點作業 | 無法進入匯入作業、D 站全檢、管理後台 |
| supervisor | 站點總覽、工程師 KPI、匯入作業、D 站全檢 | 適合現場主管支援跨站與全檢 |
| manager | 站點總覽、工程師 KPI、匯入作業、D 站全檢 | 可進行流程管理但不可進入 admin 後台 |
| admin | 全部模組 | 包含後台設定、資料工具與帳號管理 |

> 若現場人員反映無法看見匯入或 D 站頁面，請先確認其帳號角色是否為 **supervisor / manager / admin**。若無法看見管理後台，則需確認是否為 **admin**。[7] [8]

## 4. 主要作業入口

登入後若系統偵測到角色屬於管理導向，會依首頁查詢結果進行自動導流；首頁同時保留手動入口，包含「前往站點作業總覽」、「前往工程師 KPI」與「前往管理後台」。因此教育訓練時，建議先教使用者理解：系統不是固定把每個人鎖在單一站，而是讓人員依當下任務在總覽中切站支援。[1] [2] [3]

### 4.1 站點總覽頁

站點總覽是現場最常使用的工作首頁。頁面會列出各站未完成件數、今日新增件數與逾期件數，並提供「進入站點作業」按鈕。D 站會導向專用的 `/sampling` 全檢頁，其餘站點則進入 `/station/:stationCode`。頁面同時放置 KPI 摘要與 ERD 對應的作業節奏說明，幫助現場人員判斷下一個要支援的站點。[3]

| 總覽資訊 | 用途 |
| --- | --- |
| 未完成 | 判斷當前積壓量 |
| 今日新增 | 觀察當日進件壓力 |
| 逾期件數 | 快速辨識高風險站點 |
| 今日 KPI 摘要 | 協助工程師掌握當日點數與達標率 |

## 5. 站點流程總覽

系統支援 A1、A2、B、C、D、E 與待入庫（STOCK）等站點。整體上，商品可先由匯入作業建立，也可在 A1 站做補齊與到貨。A1 完成後會進入 A2；之後依品類流程設定，通常會流向 B、C、D、E，最後進入待入庫。不同品類可由管理後台設定不同站點集合，例如某些品類可能略過 A2 或 B。這代表教育訓練時，需把「標準流程」與「依品類而變的流程」分開說明。[5] [6] [8]

| 站點 | 當前定位 | 核心操作 |
| --- | --- | --- |
| 匯入作業 | 建立待處理批次 | CSV 匯入、建立 PO、設定共同到貨時間 |
| A1 點到貨 | 補齊缺漏資料並建立到貨完成節點 | 掃碼、補批號/序號/IMEI/品名 |
| A2 安裝 | 快速掃碼完成 | 掃碼後直接完工推進 |
| B 站軟測 | 補記電池檢測與非螢幕功能狀態 | 查找商品、記錄 B 站結果 |
| C 站品檢 | 補記螢幕、機身、鏡頭狀態 | 可承接並必要時修改 B 站資料 |
| D 站全檢 | 全數複核與結果確認 | 帶入 B/C 摘要、決定送 E 或返工 C |
| E 站抹除 | 抹除完成與回寫 | 掃碼後完成並回到輸入狀態 |
| 待入庫 | 最終待入庫明細 | 只顯示清單，不做額外操作 |

## 6. 匯入作業操作說明

匯入作業頁面目前限 `supervisor`、`manager`、`admin` 存取。使用者可以上傳 CSV、下載範例檔，並填寫共同欄位，例如 PO 單號、廠商與到貨時間。若 PO 單號留空，系統會自動產生。匯入資料要求每列至少填寫一個識別欄位（商品批號／商品序號／IMEI 其一），且商品分類與品牌為必填；品名可先留空，後續在 A1 補齊。[4] [8]

### 6.1 匯入前應確認的欄位

| 欄位 | 是否必填 | 說明 |
| --- | --- | --- |
| PO 單號 | 否 | 留空由系統自動產生 |
| 廠商 | 是 | 同批共用 |
| 到貨時間 | 建議填寫 | 同批共用 |
| 商品分類 | 是 | 每列必填 |
| 品牌 | 是 | 每列必填 |
| 商品批號 | 三選一至少一項 | 與序號、IMEI 任一即可 |
| 商品序號 | 三選一至少一項 | 與批號、IMEI 任一即可 |
| IMEI | 三選一至少一項 | 與批號、序號 任一即可 |
| 品名 | 否 | 可先留空，之後再補 |

### 6.2 匯入後頁面用途

匯入完成後，頁面下方會顯示「已匯入未完成點貨的採購單」，讓使用者知道哪些資料尚未進入 A1 完成點貨。這個區塊適合用在班前確認或交班核對，避免重複匯入或遺漏批次。[4]

## 7. A1 點到貨與 A2 安裝

A1 站已調整為掃碼補齊模式。現場人員在 A1 需補齊商品批號、商品序號、品名，IMEI 則改為非必填。系統會優先比對已匯入資料並補足缺漏欄位，完成後直接推進到 A2。頁面同時會顯示「目前待點貨商品分類與數量」，讓人員先看待辦分布再開始逐筆作業。[6] [8]

A2 站則是掃碼快速完工模式。掃入對應條碼後，可直接完成 A2 並推進下一站，適合追求連續大量處理的場景。[6] [8]

> A1 的必要條件是 **商品批號、商品序號、品名**。IMEI 現在不是必填，因此現場不應因 IMEI 缺漏而卡住 A1 完工。[8]

## 8. B 站與 C 站作業

B 站用於補記電池檢測與非螢幕功能狀態。頁面已調整為較寬的單欄閱讀體驗，並保留未處理表格清單。C 站則承接 B 站資料，負責記錄螢幕狀態、機身外觀與鏡頭狀態，同樣採與 B 站一致的閱讀風格。若 C 站修改了來自 B 站的內容，系統會以對應同步邏輯回寫 Google Sheet 的變更標記與結果欄位。[5] [8]

| 站點 | 主要記錄內容 | 與前站關係 |
| --- | --- | --- |
| B | 電池檢測、故障狀態 | 自身建立初版軟測結果 |
| C | 螢幕狀態、機身外觀、鏡頭狀態 | 承接 B，必要時修正 B 結果 |

## 9. D 站全檢與結果確認

D 站目前是「全檢」而非抽樣。頁面會先帶入 B 站與 C 站的文字結果，讓使用者在單一張結果卡中完成複核、必要修改與送站決策。若全檢通過，案件會送往 E 站；若不通過，則可填寫異常原因並返工回 C 站。頁面同時支援以商品批號、商品序號或產品編號搜尋，以便現場快速定位單筆案件。[5] [9]

本輪版型已調整為更接近 B 站的寬版單欄閱讀體驗，重點包含獨立搜尋卡、放寬資訊區塊、清楚的欄位分區與較明確的操作按鈕位置，目的就是降低現場判讀壓力。[9]

## 10. E 站抹除與待入庫

E 站目前支援刷碼／條碼輸入。完成抹除後，系統會自動回到待輸入狀態，方便連續處理下一台設備。待入庫頁則已改成僅顯示詳細清單，不提供額外按鈕操作，讓倉儲或交接人員可直接比對批號、序號與 IMEI。[5] [6] [8]

## 11. 工程師 KPI 頁面

工程師 KPI 頁面會呈現當日總點數、原始達成率、KPI 達標率與超標率，並用四大維度摘要解釋績效邏輯：產能達成、品質表現、時效表現與出勤公平性。其中月報採 **日均點數** 而非單純總件數，避免因出勤天數不同造成誤判。頁面另提供月度摘要，如出勤天數、月總點數、月日均點數與月平均達成率。[10]

| KPI 面向 | 說明 |
| --- | --- |
| 產能達成 | 依點數制衡量工作量 |
| 品質表現 | 抽檢不良率、返工率會列入扣分 |
| 時效表現 | 觀察逾期件數與平均處理時數 |
| 出勤公平性 | 月報以日均點數替代總件數比較 |

## 12. 管理後台操作總覽

管理後台僅限 `admin` 使用。頁面目前分為 **站點規則、產能設定、功能表設定、資料工具、帳號管理、品類設定** 六個主要分頁，並整合為單一「全部儲存」行為，儲存成功後會顯示成功提示。這代表管理者在教育訓練中必須理解：後台不是每個區塊各自存檔，而是多個設定區塊整體送出。[5] [8]

### 12.1 KPI 與分析區塊

管理後台首頁區塊可查看全員 KPI 進度，並支援開始日期、結束日期的篩選器，方便管理者查看特定期間的工程師表現。此外，也有匯入到各節點平均天數、待入庫週期等管理分析資訊，供管理者追蹤流程效率。[5] [8]

### 12.2 設定區塊

| 分頁 | 功能 |
| --- | --- |
| 站點規則 | 設定路由、下一站、返工站與啟用狀態 |
| 產能設定 | 依站點與品類設定每日產能，作為每小時產能與點數換算基礎 |
| 功能表設定 | 維護 B/C 站的故障、外觀、鏡頭等選項 |
| 帳號管理 | 建立新帳號與角色 |
| 品類設定 | 維護商品類別 × 品牌組合，並設定各品類要經過的站點 |

### 12.3 資料工具

資料工具是目前管理後台最重要的操作工具之一，包含以下功能：

| 工具 | 用途 | 使用建議 |
| --- | --- | --- |
| 匯入批次備份／還原 | 匯入前後建立快照，避免誤匯入污染 | 還原前先看預覽筆數與差異摘要 |
| 刪除指定採購單 | 清掉錯誤匯入或重複匯入的 PO | 建議先建立備份後再刪除 |
| 商品批號／序號追蹤 | 查單筆商品各站任務、時間軸、事件與耗時異常 | 用於排查卡關或跨站溝通 |

> 「刪除指定採購單」不是日常整理工具，而是處理 **匯入錯誤、重複匯入、需先清空後重傳** 的修復型操作。正式操作前，建議一定先做備份。[5]

## 13. 常見作業情境建議

### 13.1 新到貨一批商品

建議先由具權限者進入匯入作業，確認 CSV 與共同欄位後完成匯入，再由 A1 開始補齊到貨資訊。若資料量大，可先到站點總覽確認哪一站待辦最多，再安排支援人員。[3] [4] [6]

### 13.2 發現 PO 匯入錯誤或重複

先到管理後台的資料工具建立該 PO 備份，再使用「刪除指定採購單」，確認主資料已清除後重新匯入。若已造成污染，也可用還原功能回到備份狀態。[5]

### 13.3 單筆商品卡在流程中

可使用「商品批號／序號追蹤」搜尋商品批號、序號或 IMEI，查看各站任務、事件與耗時異常。若顯示特定站點長時間未前進，可直接通知對應站點處理。[5]

### 13.4 D 站判斷與前站結果不一致

D 站應直接在該筆結果卡中展開 B 或 C 的修改區，修正文字結果後決定送往 E 或返工回 C。若要返工，請填寫異常原因，讓後續人員知道返工背景。[9]

## 14. 教育訓練建議節奏

實務上建議將教育訓練分成三段。第一段先講角色與入口，讓使用者知道自己平常該從哪裡進入。第二段帶著走一次標準流程，從匯入、A1、A2、B、C、D、E 到待入庫。第三段再教管理者操作管理後台與資料工具。若一次把全部功能都講完，現場人員往往只記得自己那一站，因此分層訓練會更有效。

| 對象 | 建議訓練重點 |
| --- | --- |
| 工程師 | 站點總覽、A1/A2/B/C/E、個人 KPI |
| 主管 | 站點總覽、匯入作業、D 站全檢、跨站協調 |
| 管理者 | 管理後台六大分頁、備份還原、品類流程與 KPI 篩選 |

## 15. 已知邊界與注意事項

目前系統雖已具備多站點流程與後台工具，但仍應注意幾件事。第一，管理型頁面有角色限制，教育訓練時需先確認帳號是否已正確配置。第二，不同品類可能使用不同站點流程，因此不要假設所有商品都一定會走相同路徑。第三，資料修復型操作，例如刪除 PO 或還原備份，應由管理者依流程執行，不建議一般現場人員自行處理。第四，Google Sheet 同步雖已做非同步化，但若出現欄位驗證問題，仍應由管理者透過驗證腳本或資料工具追查。[5] [6] [8] [9]
