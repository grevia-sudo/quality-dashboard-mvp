# 回收品檢系統 MVP PRD

作者：**Manus AI**  
版本日期：2026-04-27

## 1. 文件摘要

本文件定義 **回收品檢系統 MVP** 的產品目標、角色需求、功能範圍、流程設計、資料與權限規則，以及目前版本的已知限制。本文以現行專案實作為基礎整理，適合作為後續優化、交接與需求對齊的正式依據。[1] [2] [3] [4] [5] [6] [7]

## 2. 產品背景

回收品檢流程天然具有多站點、跨角色、跨資料來源的特性。若仍依賴手工表單、零散的 Google Sheet 更新與口頭交接，容易出現資料遺漏、站點卡關、返工原因不清，以及管理者無法掌握全流程效率等問題。回收品檢系統 MVP 的核心目標，就是把商品自匯入起到待入庫為止的節點全部納入同一套系統，並同時提供現場站點作業、工程師 KPI 與管理後台能力。[1] [2] [3]

## 3. 產品目標

本產品目前的目標不是做一套通用 ERP，而是優先解決現場流程可追蹤、資料不易污染、管理者可調整流程、跨站資訊能被承接，以及 KPI 能被可視化這幾個核心問題。對現場來說，它要能減少重複輸入與查找時間；對管理者來說，它要能支援規則設定、資料修復與效能追蹤；對整體營運來說，它要能讓資料與流程更可控。[2] [3] [5]

| 產品目標 | 說明 | 目前落地方式 |
| --- | --- | --- |
| 流程數位化 | 讓匯入、到貨、各站檢測、全檢、待入庫串接 | 以多頁面站點流程與後端程序承接 |
| 跨站資訊承接 | 讓 B/C/D/E 可帶入前站結果，減少重複輸入 | B/C/D 站文字結果承接與修正 |
| 管理可控 | 讓管理者調整流程、產能與功能選項 | 管理後台六大分頁與 save-all |
| 風險可修復 | 避免誤匯入、重傳污染與資料遺失 | 備份／還原、刪除指定 PO、追蹤工具 |
| 績效可量化 | 讓工程師與管理者掌握 KPI 進度 | 個人 KPI 與全員 KPI 儀表板 |

## 4. 角色定義

目前系統中，實際對產品流程有明確影響的角色包括 `user`、`supervisor`、`manager` 與 `admin`。此外，帳號建立流程也支援 `engineer` 等角色值，但目前前端主要權限判定仍以一般使用者、管理型作業角色與 admin 管理者為主。[4] [6]

| 角色 | 主要工作 | 權限摘要 |
| --- | --- | --- |
| user | 一般站點作業與個人 KPI 查看 | 可登入、進行一般站點作業 |
| supervisor | 現場主管、跨站支援、D 站與匯入作業 | 可用匯入與 D 站全檢 |
| manager | 流程管理與現場支援 | 可用匯入與 D 站全檢 |
| admin | 系統配置、資料工具、帳號管理 | 可進入管理後台與全部模組 |

## 5. 主要使用者問題

### 5.1 現場工程師問題

現場工程師最大的痛點，通常不是功能不足，而是流程資訊分散。若每一站都要重找商品、重打資料、重問前站結果，實際效率會明顯下降。因此系統設計上強調掃碼、快速定位、前站結果承接與回到待輸入狀態，目的就是降低每站切換成本。[3] [6]

### 5.2 主管與管理者問題

主管與管理者面對的問題則是全局可視性不足。他們需要知道哪一站卡住、哪一位工程師達標率偏低、某品類是否流程過長、是否發生誤匯入與資料污染。因此產品除了站點頁，也必須提供總覽、KPI 分析、備份還原與商品追蹤功能。[3] [5]

## 6. 功能範圍

### 6.1 本期已實作範圍

| 功能模組 | 已實作內容 |
| --- | --- |
| 首頁導流 | 依角色自動導向站點總覽或管理後台，保留手動入口 |
| 站點總覽 | 顯示各站未完成、今日新增、逾期件數與 KPI 摘要 |
| 匯入作業 | CSV 匯入、PO 管理、共同到貨時間、待點貨採購單摘要 |
| A1 / A2 | A1 補齊到貨資料、A2 掃碼快速完工 |
| B / C | B 站電池與故障狀態、C 站螢幕/外觀/鏡頭狀態 |
| D 站全檢 | 承接 B/C 結果、搜尋、修改、送 E 或返工 C |
| E 站 | 掃碼抹除、完成後回到輸入狀態 |
| 待入庫 | 顯示詳細清單，不做額外按鈕操作 |
| 工程師 KPI | 個人當日與月度績效摘要 |
| 管理後台 | KPI、規則、產能、功能選項、品類設定、資料工具、帳號管理 |
| Google Sheet 同步 | 各站完成時間與人員欄位回寫、跨站歷史修補 |
| 資料修復工具 | 備份／還原、刪除指定 PO、商品追蹤 |

### 6.2 本期明確不在範圍內

目前版本沒有把系統做成對外客戶或供應商可用的平台，也不是為了取代完整 ERP。其目的仍聚焦在內部回收品檢作業與管理。另就現有上下文來看，也沒有納入部署到外部第三方平台、外部通知工作流、或完整行動版原生 App 規劃。[2] [5]

## 7. 關鍵使用流程

### 7.1 流程一：從匯入到待入庫

標準場景是由管理型角色先進行匯入，建立 PO 與待處理資料，再由 A1 補齊到貨與缺漏資訊，A2 做快速掃碼完工，後續依品類流程進到 B、C、D、E，最後進入待入庫。此流程最重要的設計要點，是 **不同品類可有不同節點集合**，因此流程並非所有品類完全一致。[4] [5] [6]

### 7.2 流程二：D 站全檢判斷

D 站全檢的產品設計目標，是把前站資訊整合到單張卡片中，降低全檢人員來回比對多頁面的負擔。D 站會帶入 B/C 的結果，並允許在必要時直接修正文字摘要。如果全檢通過，案件送往 E；若不通過，則帶著異常原因返工至 C。[7]

### 7.3 流程三：資料修復

資料修復流程主要由 admin 使用。當管理者發現 PO 匯入錯誤、重複匯入或污染時，應先建立備份，再考慮刪除指定 PO 或還原備份。若需要追查為何某商品未往下一站流動，則使用商品追蹤工具，從單筆商品的時間軸與事件紀錄切入。[5]

## 8. 產品規則與商業邏輯

### 8.1 權限規則

匯入作業與 D 站全檢只開放 `supervisor`、`manager`、`admin`；管理後台只開放 `admin`。若使用者嘗試直接進入不符合權限的頁面，前端會導回站點總覽，後端查詢也使用對應程序做保護。[4] [6] [8]

### 8.2 匯入規則

匯入時，廠商、商品分類與品牌為必要欄位；商品批號、商品序號、IMEI 至少要填一項；品名可留空。PO 單號可以自動生成。這是因為系統優先保證資料有「可辨識身份」，其餘欄位可於後續站點補足。[4] [6]

### 8.3 A1 規則

A1 目前的必要完成條件是商品批號、商品序號與品名，IMEI 非必填。此調整反映現場實際作業需求，避免因單一設備類型沒有 IMEI 或暫時無法取得 IMEI 而卡住流程。[6] [9]

### 8.4 品類流程規則

管理者可以為不同商品類別 × 品牌組合設定不同站點流程。這是 MVP 非常關鍵的能力，因為並非所有品類都需要經過 A1/A2/B/C/D/E 的完整鏈路。例如某些品類可能不需要 B 站或 A2。產品規格上，這個能力直接影響下一站與返工站的計算方式。[5] [6]

### 8.5 KPI 規則

KPI 頁面與月報邏輯已明確強調 **日均點數**、**出勤公平性** 與多維度判讀，而不是只比較總件數。品質表現與時效表現會以扣分或警示的方式被納入考量，避免人員只追求高產能而忽略返工與逾期。[3] [10]

## 9. 管理後台需求定義

### 9.1 KPI 與分析

管理者需能查看全員 KPI 進度，且支援時間範圍篩選。此需求源於管理層需要查看特定期間的績效，而不只是固定當天或當月數據。系統目前已提供開始日期、結束日期與「重設為當月」操作。[5]

### 9.2 流程與規則設定

管理者需能設定站點規則、下一站與返工站，並針對不同品類配置流程。這項需求的核心價值，在於讓流程配置不再寫死在程式裡，而是由後台維護。[5] [6]

### 9.3 產能與選項維護

管理者需能維護 A1、A2、B、C、D、E 各站點依品類的每日產能，並能維護 B/C 站的故障、外觀與鏡頭等選項。這些設定會直接影響現場畫面、點數換算與品類流程管理。[5] [6]

### 9.4 資料安全與追蹤

管理者需具備誤匯入後的修復能力，因此資料工具需同時支援備份、還原、刪除指定 PO 與單筆商品追蹤。這是 MVP 能否在現場真正被信任的關鍵條件之一。[5]

## 10. Google Sheet 整合需求

系統目前仍與 Google Sheet 保持同步關係，尤其在採購單相關欄位上。這表示產品層面的需求不只是讓資料停留在本系統內，也包括對外部試算表欄位進行時間、人員與狀態回寫。近期已完成 E 站完成時間與人員驗證、跨站歷史時間偏移修正、C 站鏡頭欄位回寫與 D 站相關欄位寫回邏輯收斂等工作。[7] [9]

| 站點／模組 | 同步需求摘要 |
| --- | --- |
| C 站 | 完成時間、螢幕/外觀/鏡頭等摘要回寫 |
| D 站 | 若有修改結果需標記 W 欄；通過送 E 時需寫回 X/Y 欄 |
| E 站 | 完成時間與執行人員回寫至 Z/AA 欄 |
| 跨站 | completedAt / businessDate 使用共用 operation time helper |

## 11. 介面與體驗原則

本系統介面偏向內部工具設計，應優先服務高頻、重複、需快速切站的使用情境。設計原則包括：總覽先看待辦與逾期、站點頁儘量單欄寬版、搜尋輸入與結果卡保持接近長度、前站結果盡量可承接、完成後應回到可持續下一筆的狀態。近期 D 站版型調整，就是依照 B 站較佳的閱讀體驗進行對齊。[3] [7]

## 12. 已知限制與風險

目前版本仍有幾項應持續留意的限制。第一，Google Sheet 仍屬外部依賴，雖已加入非同步寫回與驗證腳本，但若外部資料異常仍需管理者排查。第二，產品流程具有高度品類差異，因此教育訓練與 SOP 必須跟著品類流程設定更新。第三，資料工具屬高風險操作，若權限或流程不清楚，可能造成錯誤刪除與還原誤判。第四，既有測試中仍存在部分與本次任務無關的歷史失敗案例，因此回歸時需區分本次 targeted 驗證與全量穩定性議題。[5] [6] [9]

## 13. 成功指標建議

MVP 階段的成功不應只看系統是否能用，而應同時看現場採用率、跨站交接成本是否下降，以及管理者是否能更快排查流程異常。以下是較實際的產品指標建議：

| 指標 | 定義 | 目的 |
| --- | --- | --- |
| 匯入後 24 小時內完成 A1 比例 | 已匯入商品中，於 24 小時內完成 A1 的占比 | 觀察到貨補齊效率 |
| 各站逾期件數 | 總覽頁逾期件數累計 | 找出瓶頸站 |
| 返工率 | D 站返工回 C 站的比例 | 評估前站品質穩定性 |
| 平均單站耗時 | 商品追蹤中的各站 duration | 追蹤流程效率 |
| 備份後還原成功率 | 備份／還原工具的實際成功情況 | 驗證資料安全性 |

## 14. 後續優先規劃建議

下一階段若要延伸產品價值，建議不要平均展開所有功能，而應優先做能縮短交接成本與降低管理風險的項目。第一，可以針對 D 站、E 站與待入庫補更多互動層級測試與操作紀錄。第二，可將商品追蹤延伸為可匯出單筆歷程報表。第三，可建立更明確的管理日誌，記錄誰做了 PO 刪除、備份、還原與流程調整。[5] [7]
