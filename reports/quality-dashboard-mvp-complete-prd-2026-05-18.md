# 回收品檢系統 MVP 完整 PRD

作者：**Manus AI**

## 1. 文件摘要

本文件整理 **quality-dashboard-mvp（回收品檢系統 MVP）** 截至目前版本的完整產品需求定義，目的在於作為營運、現場、管理者、設計、工程與後續擴充的共同基準。文件內容以目前已落地的系統行為、後端規則、教育訓練手冊、管理後台配置能力，以及近期完成的 KPI 複核頁與 Google 同步修正為基礎，定義系統的產品定位、使用者角色、業務流程、功能模組、KPI 邏輯、資料規則、整合需求、風險與下一階段規劃。[1] [2] [3] [4] [5] [6]

## 2. 產品背景與問題定義

回收品檢流程的核心挑戰，在於同一件商品會橫跨多個作業站點、多人協作與多資料來源。若仍依賴 Google Sheet、口頭交接與人工追查，不但容易出現資料遺漏，也會使站點之間的責任邊界模糊，進而導致返工、逾期、KPI 爭議與後續待入庫錯漏。回收品檢系統 MVP 的產品使命，是把商品從匯入到待入庫的全生命週期收斂進同一套內部系統，並在不破壞現場節奏的前提下，提供站點作業、工程師績效、管理設定、資料修復與 Google Sheet 對帳能力。[1] [2] [5]

從產品角度來看，這不是一套對外販售的通用 ERP，也不是用來取代企業所有主資料流程的平台。它的首要價值，在於把現場高頻、重複、易遺漏且需要快速交接的回收檢驗流程數位化，並讓管理層可以看見流程瓶頸、KPI 來源、資料污染風險與站點規則是否合理。[1] [2]

## 3. 產品目標

本期產品目標聚焦於三個層次。第一個層次是 **流程可執行**，也就是每一件商品可以在系統中被建立、被辨識、被推進到下一站，並且能依商品類別走不同流程。第二個層次是 **資料可追蹤**，管理者要能看見站點任務、時間軸、返工原因、Google 同步與資料修復結果。第三個層次是 **績效可複核**，工程師與管理者不只要看 KPI 數字，還要能往下追到每件商品、每一站、每一筆事件是否真的被計分。[1] [2] [5] [6]

| 產品目標 | 需求定義 | 現行落地方式 |
| --- | --- | --- |
| 流程數位化 | 商品從匯入、A1、A2、B、C、D、E 到待入庫都可在系統內流轉 | 多站點頁面、任務狀態與下一站規則 |
| 品類差異化流程 | 不同品牌 × 品類可走不同站點集合 | 管理後台品類流程設定 |
| 現場快速作業 | 搜尋、掃碼、承接前站資訊、完成後快速回到下一筆 | A1/A2/B/C/D/E 專用站點頁 |
| 管理可配置 | 流程、產能、站點選項與權限配置可在後台維護 | admin 後台多分頁設定 |
| 資料可修復 | 匯入失誤、重複匯入、特定 PO 污染可追查與回復 | 備份／還原、刪除 PO、商品追蹤 |
| KPI 可複核 | 不只看總分，也能看每件商品在各站點的可計分狀態 | 工程師 KPI 與 KPI 複核頁 |
| Google 對帳 | 必要欄位可回寫試算表，並支援同步驗證與排錯 | 採購單同步與站點欄位回寫 |

## 4. 產品範圍

### 4.1 本期已納入範圍

本期已納入的產品能力，包含首頁導流、站點總覽、匯入作業、A1/A2/B/C/D/E/待入庫流程、工程師 KPI、管理後台、Google Sheet 採購單同步、資料工具，以及近期新增的 **KPI 複核頁**。其中 KPI 複核頁的產品價值，在於它把過去只能看總分的管理方式，延伸成可直接查單件商品、單一工程師、單一站點的計分狀態與缺漏原因。[1] [2] [5] [6]

### 4.2 本期不在範圍內

本期不包含外部供應商入口、客戶端自助查詢、完整 ERP 財務模組、第三方部署流程、公共 API 商業化開放，以及原生手機 App。現有產品仍屬內部作業平台，主要服務現場工程師、主管與管理者。[1] [2]

## 5. 目標使用者與角色權限

系統中的主要角色可分為一般作業人員與管理型角色。一般工程師的核心任務，是在站點頁完成實際檢測與查看個人 KPI。主管與管理者則承擔跨站支援、匯入作業與 D 站全檢。`admin` 則負責設定流程、維護產能、處理資料風險與管理帳號。[1] [2] [6]

| 角色 | 主要目標 | 可操作模組 | 權限原則 |
| --- | --- | --- | --- |
| user | 完成站點作業、查看個人 KPI | 站點總覽、A1/A2/B/C/E、個人 KPI | 不可進匯入、D 站全檢、管理後台 |
| supervisor | 現場協調與跨站支援 | user 全部能力 + 匯入作業 + D 站全檢 | 不可進 admin 後台 |
| manager | 流程管理與現場調度 | supervisor 同級能力 | 不可進 admin 後台 |
| admin | 系統配置、資料修復、營運複核 | 全模組 | 唯一可進管理後台與資料工具 |

> 權限策略的核心不是把人綁死在單一站點，而是確保高風險操作，例如匯入、全檢、PO 刪除、備份還原與設定維護，只能由適合的角色執行。[1] [2] [6]

## 6. 核心使用情境

### 6.1 情境一：新到貨批次進件

管理型角色先於匯入作業頁上傳 CSV，建立採購單與商品主檔。若 PO 單號留空可自動生成，但廠商、商品分類與品牌需完整，且商品批號／商品序號／IMEI 三者至少要有其一，確保身份可被辨識。匯入後，A1 站補齊到貨資訊並確認可推進欄位，A2 站做快速掃碼完工，之後依品類流程進入 B/C/D/E 或其他必要站點。[2] [6]

### 6.2 情境二：標準站點檢測與跨站承接

B 站會建立初版的軟測與故障結果；C 站承接 B 站資訊並記錄螢幕、外觀、鏡頭等狀態；D 站把 B/C 的摘要整合在單一結果卡中，讓全檢人員可以在一處判斷是送 E、返工 C 或補充修正；E 站完成抹除作業後，商品進入待入庫。這個流程的產品要求，是跨站資訊要足夠完整，避免每一站重新建立上下文。[2] [6]

### 6.3 情境三：KPI 與複核排錯

工程師可在個人 KPI 頁查看當日與月度分數摘要，管理者則可從管理後台查看全員 KPI 分析與近期新增的 KPI 複核頁。若出現「同樣看 Google 資料，為何與手算不同」的情況，管理者應能從 KPI 複核頁直接追到商品層級，看見某站是否有完成事件、是否缺 KPI 明細、是否缺分類、是否少了 target config，或是否因 D 站補寫機制尚未觸發而造成差異。[3] [4] [5]

### 6.4 情境四：資料修復與污染處理

若發生重複匯入、錯誤 PO、誤測資料或 Google 測試列污染，`admin` 需能使用後台資料工具先備份，再刪除指定採購單、還原批次、追查單筆商品，或對測試資料做清理。近期專案已實際驗證過清除測試批號、重算受影響 KPI 與同步清掉關聯資料的作業方式，代表資料安全與可逆性是正式產品需求的一部分。[4] [5]

## 7. 端到端業務流程

本系統的端到端流程不是完全固定線性流程，而是 **由品類流程設定決定實際站點集合**。也就是說，A1 → A2 → B → C → D → E → 待入庫是常見標準流程，但並非每個商品都必然經過全部站點。產品需求因此必須保證：系統可根據品牌 × 品類自動推導下一站與返工站，並在總覽、站點頁、KPI、Google 同步與追蹤頁面上保持一致。[1] [2] [6]

| 流程階段 | 主要輸入 | 主要輸出 | 產品要求 |
| --- | --- | --- | --- |
| 匯入 | CSV、PO、廠商、分類、品牌 | 待處理商品、採購單、初始流程身份 | 身份可辨識、匯入可回溯 |
| A1 | 批號／序號／品名補齊 | 到貨完成、進 A2 或下一站 | 不因 IMEI 缺漏卡關 |
| A2 | 掃碼 | 快速完工與推進 | 高頻處理效率 |
| B | 電池與功能檢測 | 初版軟測結果 | 可成為後站承接基礎 |
| C | 螢幕／外觀／鏡頭檢測 | C 站摘要與必要修正 | 回寫 Google 與修正追蹤 |
| D | 綜合複核與抽樣通過 | 送 E 或返工 C | 支援補寫 KPI 條件 |
| E | 抹除完成 | 待入庫 | 完成後回到可處理下一筆 |
| 待入庫 | 已完成商品 | 倉儲清單 | 以清單比對為主，不做多餘操作 |

## 8. 功能需求定義

### 8.1 首頁與導流

首頁需要依使用者角色決定建議入口，並保留手動入口，避免使用者被過度自動導向而失去切換權。設計原則是讓人員進站快、找功能快、理解自身可用範圍也快。[1] [2]

### 8.2 站點總覽

站點總覽必須展示各站未完成數、今日新增數、逾期件數與 KPI 摘要，作為現場班前與交班的共用視圖。D 站需導向專用全檢頁，其餘站點則進入各自站點頁。[2]

### 8.3 匯入作業

匯入作業需支援 CSV 上傳、範本下載、PO 建立與共同欄位套用。規格上需保證每列至少有一個身份欄位，商品分類與品牌為必要欄位，PO 可自動生成，品名可於後續補齊。[2] [6]

### 8.4 A1 / A2

A1 的產品要求是「低阻塞補齊」，必要完成條件為商品批號、商品序號與品名，IMEI 不應再成為硬性阻擋條件。A2 則為「極簡快速完成」，掃碼即能高頻處理，適合作業節奏快的站點。[2] [6]

### 8.5 B / C

B 站與 C 站的需求重點在於資料承接、閱讀流暢與修正可追溯。B 站記錄電池與故障狀態，C 站記錄螢幕、機身、鏡頭，並在必要時修正前站結果。C 站同步邏輯需能把本地已有的 C 區塊資料完整刷新到 Google 採購單，避免只更新部分欄位造成誤判。[4] [5]

### 8.6 D 全檢

D 站需提供單卡式綜合判讀視圖，承接 B/C 結果、支援搜尋、修改與送站決策。全檢通過應能送 E，不通過應可返工回 C。KPI 上，D 站與一般站點不同，需把 `sampling_pass` 納入 KPI 補寫條件，而非僅依一般完成事件判定。[3] [6]

### 8.7 E 抹除與待入庫

E 站需支援刷碼與最小必要照片資料，完成後回到輸入狀態。待入庫頁則應以明細檢視為主，不增加額外操作干擾。[2] [6]

### 8.8 工程師 KPI

工程師 KPI 頁必須同時呈現當日總點數、達標率、超標率與月度摘要。月度計算口徑應以 **日均點數** 為核心，而不是單純總件數，以避免因出勤天數不同而不公平。[1] [2]

### 8.9 管理後台

管理後台至少需涵蓋站點規則、產能設定、功能表設定、帳號管理、品類設定、資料工具與 KPI 分析。設定行為目前採「全部儲存」，因此產品必須讓管理者清楚知道變更是整體送出，而不是單區塊即時生效。[2] [6]

### 8.10 KPI 複核頁

KPI 複核頁是近期新增的核心後台功能。它的需求不是只再做一張報表，而是讓管理者可用商品層級方式查看 **A1、A2、B、C、D、E** 各站是否已有可計分事件、是否已產生 KPI 明細、缺漏原因為何、對應工程師為誰，以及可依批號、序號、IMEI、品名、工程師、站點與日期區間做篩選。這是解決 KPI 爭議、Google 對帳與現場手算落差的關鍵功能。[3] [5]

## 9. KPI 與計分需求

### 9.1 計分原則

系統的 KPI 計分並不是單純以 Google Sheet 上看到的分類件數直接相加，而是先把站點事件與商品主檔回對，再依有效的 `categoryId`、`subtype` 與對應站點的 active `productivity_target_configs` 換算每件分數。也因此，Google 看到有完成事件，並不代表系統一定已成功寫入 KPI 明細。[3] [4] [5]

### 9.2 計分前置條件

目前產品邏輯下，會實際影響 KPI 是否可寫入的主要條件包括：商品需具備可計分事件、需能解析出有效的 `categoryId / subtype`、需找得到對應站點的 active target config，且 D 站需滿足抽樣通過補寫機制。相對而言，`poNumber` 與 `vendorName` 並不是直接的 KPI gate；但若商品分類與品牌缺漏，通常會使 `categoryId` 無法解析，進而導致該件雖有事件卻得不到 KPI 明細。[3] [4] [5]

| 條件 | 是否直接影響 KPI 寫入 | 說明 |
| --- | --- | --- |
| 站點完成事件 | 是 | A1/A2/B/C/E 以完成事件為主 |
| D 站抽樣通過 | 是 | D 站需走補寫機制 |
| 有效 categoryId / subtype | 是 | 找不到分類就找不到 target config |
| active target config | 是 | 無對應站點目標值就無法換算分數 |
| 採購單號 | 否 | 不作為直接 gate |
| 廠商 | 否 | 不作為直接 gate |
| Google baseline 關聯 | 是 | 不在基準範圍內的事件可被視為不可計分 |

### 9.3 KPI 口徑爭議與系統現況

近期核對結果顯示，目前系統對 **智慧型手機 × Apple** 的 active 目標值採用 **C = 159、E = 400**，而非部分人工手算使用的 **C = 160、E = 300**。但實務上，造成系統分數與人工手算差距較大的主因，通常不是 159/160 或 300/400 這類單一目標值差異，而是 **Google 的分類件數統計口徑** 與 **系統以商品主檔分類回對後才計分的口徑** 並不完全相同。[3] [5]

### 9.4 KPI 複核需求延伸

產品層面上，KPI 複核頁不應只顯示「未寫入 KPI 明細」，而應持續演進為能夠區分：缺分類、缺 target config、缺 Google baseline、僅有完成事件尚未補寫、D 站未達補寫條件、E 站缺必要照片資料等不同原因。這樣才能真正縮短排查時間。[3] [5]

## 10. Google Sheet 整合需求

系統目前仍需與 Google 採購單維持雙向關聯。這包含採購單列的定位、各站完成時間與工程師欄位回寫，以及必要站點摘要欄位的刷新。近期實作已驗證 C 站同步最小修正：只要本地已有任何 C 站資料，就應強制刷新整個 C 區塊欄位；同步 worker 亦必須在標記 success 前再次驗證，避免本地已有 C 完成資料但 Google 列仍空白時被錯判為成功。[4] [5]

產品上也必須明訂一條重要規則：**若要做會寫回 Google 的驗證，只能使用採購單測試資料，不得污染正式資料。** 這項規則已在近期任務中反覆驗證，且與資料工具、測試資料清理與 KPI 對帳直接相關。[4] [5]

## 11. 資料模型與關聯需求

從產品需求角度，系統至少需穩定維護五類核心資料：商品主檔、站點任務、站點事件、KPI 明細與設定資料。商品主檔保存身份、分類、流程上下文與採購單定位；站點任務與事件記錄每站狀態與作業歷史；KPI 明細則把可計分事件轉成可稽核分數；設定資料則包含站點流程、功能選項與站點目標值。[1] [2] [6]

| 資料域 | 主要用途 | 對應產品功能 |
| --- | --- | --- |
| Products | 商品身份、分類、採購單對應、流程上下文 | 搜尋、追蹤、站點流轉、KPI 分類 |
| Station Tasks | 商品在各站的任務狀態 | 總覽、站點待辦、返工與推進 |
| Station Events | 實際發生的作業事件 | KPI、追蹤、時間軸、Google 對帳 |
| Productivity Score Details | 每件商品、每站、每工程師的點數明細 | 工程師 KPI、管理 KPI、複核頁 |
| Target / Category Configs | 站點目標值、品類流程、功能選項 | 設定維護、換算邏輯 |

## 12. 管理後台需求

管理後台不是單純的設定頁，而是產品營運控制台。其需求可以分成三個層次。第一層是 **規則維護**，包括站點流程、返工邏輯、品類對應與目標值設定。第二層是 **營運觀測**，包括全員 KPI、流程效率、匯入到各節點的時間分布。第三層是 **風險控制**，包括帳號管理、備份還原、刪除 PO、商品追蹤與 KPI 複核。[2] [5] [6]

由於目前子功能頁設計偏好是「每頁只顯示該功能內容，避免上下滑動看到其他頁面」，後續後台資訊架構應持續朝分頁清楚、單一任務聚焦的方向演進。[2] [5]

## 13. 介面與體驗原則

本系統屬內部高頻操作工具，介面決策應優先服務速度、可讀性與低認知負擔。站點頁需讓使用者能快速找到下一筆、看懂前站結果、完成後回到可繼續輸入的狀態。管理頁則應讓使用者能快速定位例外狀況，而不是先閱讀大量摘要後再下鑽。近期 D 站寬版卡片化與 KPI 複核頁的商品層級視圖，都是沿著這個原則推進。[2] [5]

## 14. 非功能需求

### 14.1 權限與安全

所有受保護功能都需經過登入與角色保護。高風險操作需明確限制於 `admin` 或管理型角色，並保證前後端都做保護，而不是只靠前端隱藏按鈕。[1] [6]

### 14.2 可追溯性

每一筆商品都應能透過批號、序號或 IMEI 被追到其任務、事件、KPI 與同步狀態。這是處理卡件、爭議與資料污染的必要條件。[4] [5]

### 14.3 測試與回歸

關鍵流程至少應對站點流轉、Google 同步、KPI 補寫、KPI 複核頁查詢與管理後台導覽提供定向測試。近期已完成 C 站同步 guard、KPI 全站寫入驗證與 KPI 複核頁測試，代表這些已列入產品穩定性範圍，而非一次性修補。[3] [4] [5]

### 14.4 資料污染防護

任何會接觸 Google 正式採購單的驗證流程，都需使用明確測試批號或測試列，並具備清理與回復機制。產品應把這視為正式 SOP，而非臨時習慣。[4] [5]

## 15. 已知限制與風險

目前系統仍存在若干產品風險。第一，Google Sheet 仍屬外部依賴，若試算表欄位、列位移或人工編輯方式改變，仍可能影響同步結果。第二，KPI 仍有「Google 分類件數」與「系統商品主檔分類」之間的口徑落差，會造成人工手算與系統得分不一致。第三，高風險資料工具雖已具備，但若缺少操作記錄與審核，仍可能引發誤刪或還原判斷錯誤。第四，不同品類走不同站點的設定雖然是產品優勢，但也代表教育訓練與 SOP 必須持續更新，否則現場容易以為所有商品都應走同一路徑。[3] [4] [5]

## 16. 成功指標

產品成功不應只看「系統能不能用」，而應看它是否讓現場更快、管理更清楚、爭議更少。建議持續追蹤以下指標，作為後續迭代優先級的依據。[1] [2] [5]

| 指標 | 定義 | 目的 |
| --- | --- | --- |
| A1 24 小時完成率 | 匯入後 24 小時內完成 A1 的商品占比 | 觀察到貨處理效率 |
| 各站逾期件數 | 站點總覽的逾期積壓數 | 找出瓶頸站 |
| D 返工率 | D 站返工回 C 的比例 | 反映前站品質與全檢壓力 |
| KPI 缺漏率 | 有完成事件但無 KPI 明細的占比 | 反映分類與設定完整度 |
| Google 同步異常率 | 本地完成但未成功回寫的事件占比 | 反映對帳穩定性 |
| 資料修復成功率 | 備份、還原、PO 刪除與清理操作的成功率 | 反映資料治理成熟度 |

## 17. 後續規劃建議

下一階段建議優先推進三類能力。第一類是 **KPI 對帳閉環**，例如在 KPI 複核頁增加更明確的缺漏原因標籤、CSV 匯出、差異報告與直接回補入口。第二類是 **資料治理強化**，例如加入操作審計日誌、測試資料 TTL 或清理工具，以及更明確的 Google 測試列策略。第三類是 **流程效率深化**，例如讓管理者可直接觀察哪一站最常漏記分、哪個品類最常返工，以及匯入到待入庫的全鏈路耗時拆解。[3] [4] [5]

## 18. 結論

回收品檢系統 MVP 的核心價值，不在於做出最多頁面，而在於把現場作業、站點流轉、KPI 計分、Google 對帳與資料修復這五件事收斂到同一套可運營、可排錯、可追溯的系統中。從目前版本來看，系統已具備多站點作業、角色權限、後台設定、KPI 複核與資料修復的基礎能力；接下來的產品演進重點，應放在進一步縮小「Google 可見資料」與「系統最終計分資料」之間的口徑差距，並持續提升管理者對單件商品與單站點異常的可觀測性。[1] [2] [3] [4] [5] [6]
