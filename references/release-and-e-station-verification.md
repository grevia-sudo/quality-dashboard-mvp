# 發布版本與 E 站正式環境驗證說明

更新時間：2026-05-11

## 管理後台發布時如何避免沿用舊版本

近期出現過「預覽已正常，但正式發布後仍看到舊狀態」的原因，不是新修正沒有存在於專案，而是發布時若選到較早的 checkpoint，就會把正式站點回退到當時的內容。因此，後續若要發布管理後台相關修正，請以**最新 checkpoint**作為發布來源，不要沿用先前較舊的檢查點。

目前可直接辨識的安全做法如下。

| 情境 | 建議做法 |
| --- | --- |
| 只是想查看最新修正 | 先在預覽確認畫面正確，再從最新 checkpoint 發布 |
| 發布前不確定版本對不對 | 以最新 checkpoint 的時間與說明為準，避免選到更早版本 |
| 發布後發現畫面變舊 | 先檢查是否誤發佈舊 checkpoint，再重新以最新 checkpoint 發布 |

## E 站正式環境上傳通道說明

E 站正式環境目前已不再依賴 sandbox 內的 gws 指令通道，而是改成部署環境可用的**內建 storage 通道**。這樣做的目的，是避免正式站點因執行環境沒有 gws CLI 而出現 `spawn gws ENOENT`，導致照片無法完成上傳。

正式流程的預期行為如下。

| 驗證項目 | 預期結果 |
| --- | --- |
| E 站拍攝正面與反面照片後完成流程 | 畫面可成功完成 E 站，不會被卡在 CLI 缺失 |
| 照片上傳後背景處理 | 系統先完成站點推進，再由背景程序補寫照片連結 |
| 採購單照片欄位 | 稍後可看到正面、反面照片連結落到對應欄位 |

## 請使用者協助的實機驗證

請在正式環境用一筆真實可操作資料，完整跑一次 E 站正面／反面照片流程，重點確認三件事：第一，E 站頁面可順利完成；第二，照片有成功保存；第三，採購單對應照片連結會在背景同步後落檔。若其中任一步驟仍異常，再回報當下商品批號或時間點，即可繼續追查。
